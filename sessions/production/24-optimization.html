The Economics of Agentic AI

A production agent that runs thousands of tasks per day can easily accumulate hundreds or thousands of dollars in API costs. Unlike a single-turn chatbot, agents iterate: they call tools, reason about results, plan next steps, and sometimes loop tens of times before completing a task. Each iteration burns tokens. Latency compounds too — a 10-step chain where each step takes 2 seconds adds 20 seconds of wall-clock time. Optimization is therefore not optional: it determines whether your agent is economically viable and whether users will tolerate waiting for it.

This session covers five major levers for reducing cost and latency without sacrificing agent quality: token budget management, semantic caching, intelligent model routing, parallel tool execution, and prompt compression. Used together, these techniques routinely cut costs by 50–80% and halve end-to-end latency.

Token Budget Management

The most direct lever is controlling how many tokens flow through the system. Every token costs money and takes time. Practical strategies include:

  • Hard token limits per step — set max_tokens on each LLM call to prevent runaway responses.
  • Context window hygiene — trim conversation history aggressively; most agents only need the last N turns plus a compressed summary of earlier turns.
  • Output format constraints — instruct the LLM to reply in JSON or a structured template instead of verbose prose, then parse the structured output.
  • Observation summarization — when a tool returns a 10 000-token web page, extract only the relevant 200-token excerpt before feeding it back into the context.
class BudgetedAgent:
    def __init__(self, llm, max_tokens_per_step=512, max_total_tokens=8000):
        self.llm = llm
        self.max_tokens_per_step = max_tokens_per_step
        self.max_total_tokens = max_total_tokens
        self.tokens_used = 0

    def trim_history(self, history, keep_last=4):
        """Keep recent turns + summarise the rest."""
        if len(history) <= keep_last:
            return history
        old = history[:-keep_last]
        summary = self.llm.invoke(
            f"Summarise these turns in ≤80 words: {old}",
            max_tokens=100
        )
        return [{"role": "system", "content": f"Earlier context: {summary}"}] + history[-keep_last:]

    def call(self, messages):
        if self.tokens_used >= self.max_total_tokens:
            raise RuntimeError("Token budget exhausted")
        trimmed = self.trim_history(messages)
        response = self.llm.invoke(trimmed, max_tokens=self.max_tokens_per_step)
        # estimate token usage (real impl uses response.usage)
        self.tokens_used += self.max_tokens_per_step
        return response

Semantic Caching

Many agent invocations ask semantically similar questions: "What is the capital of France?", "Tell me France's capital", and "Paris is the capital of which country?" all map to the same factual answer. A semantic cache embeds each query, stores it in a vector database, and returns the cached response when a new query's embedding is within a cosine similarity threshold (typically 0.95+). This avoids an LLM call entirely.

Key insight: Semantic caches are most effective for knowledge-retrieval steps (tool outputs, factual questions) and less effective for creative or context-dependent generation steps.
import hashlib
from sentence_transformers import SentenceTransformer
import numpy as np

class SemanticCache:
    def __init__(self, similarity_threshold=0.95):
        self.model = SentenceTransformer("all-MiniLM-L6-v2")
        self.cache = []  # list of (embedding, response)
        self.threshold = similarity_threshold
        self.hits = 0
        self.misses = 0

    def _embed(self, text):
        return self.model.encode(text, normalize_embeddings=True)

    def get(self, query):
        q_emb = self._embed(query)
        for emb, response in self.cache:
            similarity = float(np.dot(q_emb, emb))
            if similarity >= self.threshold:
                self.hits += 1
                return response
        self.misses += 1
        return None

    def set(self, query, response):
        emb = self._embed(query)
        self.cache.append((emb, response))

    @property
    def hit_rate(self):
        total = self.hits + self.misses
        return self.hits / total if total > 0 else 0.0

Intelligent Model Routing

Not every step in an agent pipeline requires the most powerful (and expensive) model. A typical agent workflow has easy steps (intent classification, output formatting, tool-call parsing) and hard steps (complex reasoning, code generation, multi-step planning). Routing cheap steps to a smaller, faster model and hard steps to a frontier model can cut costs dramatically.

def route_model(task_type: str) -> str:
    """Return model name based on task complexity."""
    easy_tasks = {"classify", "format", "parse_tool_call", "summarize_short"}
    medium_tasks = {"summarize_long", "extract_entities", "qa_simple"}
    hard_tasks = {"plan", "reason_multi_step", "generate_code", "critique"}

    if task_type in easy_tasks:
        return "gpt-4o-mini"
    elif task_type in medium_tasks:
        return "claude-3-haiku-20240307"
    else:
        return "claude-sonnet-4-6"

class RoutingAgent:
    def __init__(self, clients: dict):
        self.clients = clients  # {"gpt-4o-mini": client1, ...}

    def run_step(self, task_type: str, prompt: str) -> str:
        model = route_model(task_type)
        client = self.clients[model]
        return client.invoke(prompt)

Parallel Tool Execution

When an agent needs results from multiple independent tools — for example, fetching stock price, news headlines, and analyst ratings simultaneously — running them sequentially adds unnecessary latency. Parallelising tool calls with asyncio or a thread pool drops total latency from sum-of-steps to max-of-steps.

import asyncio

async def fetch_all(tools, query):
    tasks = [tool.arun(query) for tool in tools]
    results = await asyncio.gather(*tasks, return_exceptions=True)
    return {tool.name: r for tool, r in zip(tools, results)}

Prompt Compression

LLMLingua and similar techniques use a smaller LM to score the importance of each token in a long prompt and drop low-importance tokens while preserving semantic meaning. In practice, prompts can be compressed to 30–50% of their original length with less than 5% degradation in downstream task quality. This is especially useful for RAG contexts that prepend large document chunks.

from llmlingua import PromptCompressor

compressor = PromptCompressor(model_name="microsoft/llmlingua-2-bert-base-multilingual-cased-meetingbank")

def compress_context(long_prompt: str, target_ratio: float = 0.4) -> str:
    result = compressor.compress_prompt(
        long_prompt,
        ratio=target_ratio,
        force_tokens=["###", "Question", "Answer"]
    )
    return result["compressed_prompt"]

Putting It Together: Optimization Pipeline

A fully optimised agent applies all five levers in sequence:

  1. Check the semantic cache before any LLM call.
  2. Compress retrieved context before prepending it.
  3. Route each reasoning step to the cheapest capable model.
  4. Execute independent tool calls in parallel.
  5. Enforce token budgets at every LLM invocation and trim history aggressively.

Add Semantic Caching

Wrap LLM calls with a semantic cache. Replay the same 10 questions and observe the cache hit rate and token savings.

import numpy as np

class SemanticCache:
    def __init__(self, threshold=0.92):
        try:
            from sentence_transformers import SentenceTransformer
            self.model = SentenceTransformer("all-MiniLM-L6-v2")
        except ImportError:
            self.model = None
        self.store = []
        self.threshold = threshold
        self.hits = 0
        self.misses = 0

    def _embed(self, text):
        if self.model:
            return self.model.encode(text, normalize_embeddings=True)
        # Fallback: hash-based (no semantic similarity)
        import hashlib
        h = int(hashlib.md5(text.encode()).hexdigest(), 16) % 1000
        arr = np.zeros(10)
        arr[h % 10] = 1.0
        return arr

    def get(self, query):
        q_emb = self._embed(query)
        for emb, response in self.store:
            if float(np.dot(q_emb, emb)) >= self.threshold:
                self.hits += 1
                return response
        self.misses += 1
        return None

    def set(self, query, response):
        self.store.append((self._embed(query), response))

cache = SemanticCache()
counter2 = TokenCounter()
MOCK_RESPONSES = {q: f"Answer to: {q}" for q in QUESTIONS}

for q in QUESTIONS:
    cached = cache.get(q)
    if cached:
        print(f"  CACHE HIT: {q[:40]}")
    else:
        # Simulate LLM call
        counter2.total_prompt += random.randint(400, 600)
        counter2.total_completion += random.randint(100, 200)
        cache.set(q, MOCK_RESPONSES[q])
        print(f"  CACHE MISS: {q[:40]}")

print(f"\nWith cache — tokens: {counter2.total}")
print(f"Cache hit rate: {cache.hits}/{cache.hits+cache.misses} = {cache.hits/(cache.hits+cache.misses):.0%}")
print(f"Token reduction: {(1 - counter2.total/counter.total):.0%}")

Add Model Routing

Classify each step as easy or hard and route accordingly. Measure the additional cost savings from cheaper models.

MODEL_COSTS = {
    "gpt-4o":      {"prompt": 5e-6,   "completion": 15e-6},
    "gpt-4o-mini": {"prompt": 0.15e-6, "completion": 0.6e-6},
}

def classify_question(q: str) -> str:
    """Simple heuristic: factual questions → easy, analytical → hard."""
    factual_keywords = ["what is", "who is", "when was", "which", "how many"]
    if any(q.lower().startswith(k) for k in factual_keywords):
        return "easy"
    return "hard"

def cost_for_step(model: str, prompt_tokens: int, completion_tokens: int) -> float:
    c = MODEL_COSTS[model]
    return prompt_tokens * c["prompt"] + completion_tokens * c["completion"]

total_cost_routed = 0.0
cache3 = SemanticCache()

for q in QUESTIONS:
    cached = cache3.get(q)
    if cached:
        continue  # free
    difficulty = classify_question(q)
    model = "gpt-4o-mini" if difficulty == "easy" else "gpt-4o"
    pt = random.randint(400, 600)
    ct = random.randint(100, 200)
    total_cost_routed += cost_for_step(model, pt, ct)
    cache3.set(q, f"[{model}] Answer to: {q}")
    print(f"  [{difficulty.upper():4}] {model:15} — {q[:35]}")

baseline_cost = counter.estimated_cost_usd
print(f"\nBaseline cost:      ${baseline_cost:.5f}")
print(f"Routed+cached cost: ${total_cost_routed:.5f}")
print(f"Total saving:       {(1 - total_cost_routed/baseline_cost):.0%}")

Parallel Tool Calls

Rewrite the tool-calling step to run multiple tools concurrently and measure latency reduction.

import asyncio
import time

async def fake_tool(name: str, query: str, delay: float) -> dict:
    await asyncio.sleep(delay)
    return {"tool": name, "result": f"Result from {name} for '{query}'"}

async def run_tools_parallel(query: str):
    tools_config = [
        ("search",    0.8),
        ("news",      0.6),
        ("calculator", 0.2),
    ]
    tasks = [fake_tool(name, query, delay) for name, delay in tools_config]
    return await asyncio.gather(*tasks)

def run_tools_sequential(query: str):
    delays = [0.8, 0.6, 0.2]
    results = []
    for name, delay in zip(["search", "news", "calculator"], delays):
        time.sleep(delay)
        results.append({"tool": name, "result": f"Result from {name}"})
    return results

# Sequential timing
t0 = time.time()
run_tools_sequential("AI agents")
sequential_time = time.time() - t0

# Parallel timing
t0 = time.time()
asyncio.run(run_tools_parallel("AI agents"))
parallel_time = time.time() - t0

print(f"Sequential: {sequential_time:.2f}s")
print(f"Parallel:   {parallel_time:.2f}s")
print(f"Speedup:    {sequential_time/parallel_time:.1f}x")
print(f"Latency reduction: {(1 - parallel_time/sequential_time):.0%}")
