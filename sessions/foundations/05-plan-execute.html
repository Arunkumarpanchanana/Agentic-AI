Plan-and-Execute Agents

Separate planning from execution for more reliable long-horizon task completion with replanning on failure.

35 min Foundations Hands-on Lab 5 Quiz Questions

Beyond ReAct: Why Separate Planning from Execution?

ReAct agents plan and execute interleaved — each Thought decides the next single action. This works well for short tasks but struggles with complex, multi-step goals where maintaining a global view matters. Plan-and-Execute architectures solve this by separating two concerns: a Planner that thinks about the whole task upfront, and an Executor that focuses on completing individual subtasks.

Key separation: The Planner sees the forest; the Executor sees the trees. This division lets you use a powerful (expensive) model for planning and a faster (cheaper) model for execution.

The Plan-and-Execute Architecture

  1. Plan: Given a goal, the Planner LLM generates a structured task graph — an ordered list of subtasks with dependencies.
  2. Execute: The Executor processes each subtask in dependency order, calling tools as needed.
  3. Observe: Results of each subtask are stored and passed to dependent subtasks.
  4. Replan (if needed): If a subtask fails or produces unexpected results, the Planner is invoked again to adjust the remaining plan.
  5. Synthesise: Once all subtasks complete, the Planner synthesises results into a final answer.

Task Graphs and Dependencies

Unlike a flat list of steps, a task graph (directed acyclic graph, DAG) captures which subtasks can run in parallel and which must wait for others. For example, researching two independent topics can happen in parallel, but writing a synthesis requires both to be done first.

Goal: "Write a market analysis report on EV adoption"

Task Graph:
  T1: Research global EV sales trends     ─┐
  T2: Research top EV manufacturers        ├─► T5: Synthesise findings
  T3: Research government incentives      ─┤
  T4: Research infrastructure challenges  ─┘    └─► T6: Write final report

T1-T4 can run in parallel. T5 depends on T1-T4. T6 depends on T5.

Plan-and-Solve Prompting

Wang et al. (2023) introduced Plan-and-Solve prompting as a zero-shot alternative. Instead of few-shot examples, the prompt instructs the model to: (1) devise a plan, (2) carry out the plan. This simple addition to chain-of-thought significantly reduces calculation errors and missing-step mistakes.

# Plan-and-Solve prompt template
PLAN_AND_SOLVE_PROMPT = """

Then, let's carry out the plan and solve the problem step by step.

"""

LangChain Plan-and-Execute Implementation

from langchain_openai import ChatOpenAI
from langchain.chains import LLMChain
from langchain.prompts import PromptTemplate
from langchain.tools import Tool
from langchain_community.tools import DuckDuckGoSearchRun
import json

# --- Planner ---
PLANNER_PROMPT = PromptTemplate(
    input_variables=["goal"],
    template="""You are a planning agent. Given a goal, break it into 3-5 concrete subtasks.
Return a JSON list of objects with keys: id, description, depends_on (list of ids).

Goal: {goal}

Return only valid JSON, no explanation:"""
)

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
planner_chain = LLMChain(llm=llm, prompt=PLANNER_PROMPT)

# --- Executor ---
search = DuckDuckGoSearchRun()

EXECUTOR_PROMPT = PromptTemplate(
    input_variables=["task", "context"],
    template="""Complete this research task. Use the context from previous tasks if relevant.

Task: {task}
Context from previous tasks: {context}

Provide a concise, factual response:"""
)
executor_chain = LLMChain(llm=llm, prompt=EXECUTOR_PROMPT)

# --- Orchestrator ---
def plan_and_execute(goal: str) -> dict:
    print(f"Planning for goal: {goal}\n")
    # Step 1: Plan
    raw_plan = planner_chain.run(goal=goal)
    try:
        tasks = json.loads(raw_plan)
    except json.JSONDecodeError:
        tasks = [{"id": "1", "description": goal, "depends_on": []}]

    print("Plan:")
    for t in tasks:
        print(f"  [{t['id']}] {t['description']} (depends on: {t['depends_on']})")

    # Step 2: Execute in dependency order
    results = {}
    for task in tasks:
        deps_context = "\n".join([f"Task {d}: {results.get(d, 'pending')}" for d in task.get("depends_on", [])])
        result = executor_chain.run(task=task["description"], context=deps_context or "None")
        results[task["id"]] = result
        print(f"\n✓ Task {task['id']} complete: {result[:100]}...")

    return results

results = plan_and_execute("Research the pros and cons of Python vs Go for backend development")

Replanning on Failure

A key advantage of Plan-and-Execute over ReAct is structured replanning. When a subtask fails, instead of the agent getting confused mid-stream, you invoke the planner again with the failure context:

def execute_with_replan(goal, tasks, max_replan=2):
    results = {}
    replan_count = 0
    remaining_tasks = tasks[:]
    while remaining_tasks:
        task = remaining_tasks.pop(0)
        try:
            result = execute_task(task, results)
            results[task["id"]] = result
        except Exception as e:
            if replan_count >= max_replan:
                raise RuntimeError(f"Max replanning attempts reached: {e}")
            print(f"Task {task['id']} failed: {e}. Replanning...")
            new_plan = replan(goal, results, str(e))
            remaining_tasks = new_plan  # replace remaining tasks with new plan
            replan_count += 1
    return results
When to use Plan-and-Execute vs ReAct: Use ReAct for tasks under ~5 steps where the path is unpredictable. Use Plan-and-Execute for tasks with 5+ steps where upfront planning adds value, or where you need to run subtasks in parallel.

Build the Executor and Run

results = {}
for task in plan:
